# 架构设计

## 领域模型

`Node` 描述节点坐标、类型、一次性收益、完成状态、可重复属性和出口属性；`Edge` 描述有向徒步连接及成本。`MovementRule` 用位移向量集合、行动力成本、剩余次数和越过未完成节点的许可描述加工品，求解器不识别任何特定加工品名称。

`MapState` 聚合节点、道路和移动规则，`GameState` 保存当前位置、行动力、已完成节点及加工品剩余次数。`RouteStep` 和 `RouteResult` 是不可变的求解输出语义；结果包含结构化失败原因。

所有输入领域对象均为冻结 dataclass，集合在边界处复制为不可变表示。求解过程只创建内部状态，不修改调用方对象。

## DP 状态

记忆化搜索键为：

```text
(current_node, remaining_action_points, completed_mask, movement_uses)
```

节点按 ID 稳定排序后映射到位掩码。`movement_uses` 是按规则 ID 稳定排序的剩余次数元组。到达出口是成功终态；资源耗尽且没有成功后继是失败终态。

## 状态转移

每个状态生成两类后继：

1. 徒步：只沿配置的有向边移动，扣除 `walking_cost`。若一条几何线段跨过仍未完成的中间节点，则拒绝该转移。
2. 加工品：将规则中的每个位移向量应用于当前坐标；存在目标节点、行动力和次数足够时可转移。能否跨过未完成节点由规则配置决定。

首次落到未完成节点时将相应位写入 `completed_mask` 并获得一次收益；再次访问不再得分。所有移动成本严格为正，因此搜索不会形成同资源状态的零成本环。

## 评分规则

只比较能够抵达出口的候选路线，依次最大化：

1. 总收益；
2. 剩余行动力；
3. 加工品剩余价值，即最小化总使用次数；
4. 由起点、终点、移动方式和成本组成的路线签名字典序。

最后一项不表达游戏价值，只保证跨运行和跨输入排列的稳定复现。

## 复杂度

设节点数为 `N`、初始行动力为 `A`，第 i 种加工品可用次数为 `U_i`。状态数上界为：

```text
O(N × (A + 1) × 2^N × Π(U_i + 1))
```

每个状态枚举当前节点的出边和所有规则向量；几何越过检查当前为 `O(N)`。因此该精确算法适合第一阶段规模有限的地图。后续若真实地图规模要求更高，可在不改变领域接口的前提下加入支配剪枝、上界估计或分层搜索。

## 解耦原因

规划核心只消费已结构化的地图和状态，不了解截图像素、OCR、设备连接、点击坐标或界面控件。这使算法可以用纯 JSON 确定性测试，也避免识别误差和设备副作用污染求解逻辑。未来 UI、截图识别与 MaaFramework 都应作为适配层：把外部信息转换为领域模型，再消费 `RouteResult`。这种边界允许各层独立演进、替换与审计。
